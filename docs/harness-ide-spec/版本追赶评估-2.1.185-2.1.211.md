# 版本追赶评估 · Claude Code 2.1.185 → 2.1.211

- 基准：功能版本对照表校对基准日 2026-06-20，覆盖到 2.1.183。本评估把 2.1.185-2.1.201、2.1.202-2.1.211 两批官方 changelog 逐条过了一遍，并对代码做了版本敏感点扫描（file:line 均为真实引用）。
- 评估日：2026-07-16。Claude Code 最新版 2.1.211（2026-07-15 发布）。
- 结论一句话：**要更新，而且不只是"缺新功能"——有四处存量逻辑在 2.1.211 语义下已经是错的**（价格表、配置分层、always-allow 存储位、MCP 配置路径），先修错的，再吃后台化带来的产品红利。

## A. 已经"算错/显示错"的（P0 · 正确性问题）

### A1. 模型价格表：四行错三行半

`shared/data/model-pricing.ts:18-24`（`PRICING_UPDATED = '2026-06-21'`），对照官方权威价（USD / 百万 token，2026-06-24）：

| 表内行 | 表内价（in/out） | 实际价 | 偏差 |
|---|---|---|---|
| `/opus/i` | 15 / 75 | Opus 4.8/4.7/4.6 全系 **5 / 25** | **高估 3 倍**（15/75 是 Opus 3 时代的价） |
| `/fable/i` | 3 / 15（注释"按 sonnet 估"） | 7 月 7 日官方公布 **10 / 50** | **低估 3 倍+**，注释里的"未公布"已失效 |
| `/haiku/i` | 0.8 / 4 | Haiku 4.5 为 **1 / 5** | 低估 20% |
| `/sonnet/i` | 3 / 15 | Sonnet 4.6 对；**Sonnet 5 促销期（至 2026-08-31）为 2 / 10**，9/1 起回 3/15 | 促销期高估 |

连锁影响：
- Token Usage 面板（spec017）的所有成本数字、"按 Sonnet 价格重算 Opus token 能省多少"的调优建议，全部失真。Opus 高估 3 倍 + Sonnet 促销没算，"切模型能省 X 美元"这个卖点数字现在是错的。
- `electron/services/session/usage-advisor.ts:30` 硬编码 `'claude-sonnet-4-6'` 作为省钱建议目标——2.1.197 起默认模型已是 `claude-sonnet-5`（促销期还更便宜），建议对象过时。
- `src/pages/Models.tsx:18-23` `CLAUDE_MODELS` 的 sonnet 角色映射 `claude-sonnet-4-6`，缺 `claude-sonnet-5` 条目。

修法建议：在关键字行之前加显式条目（`claude-sonnet-5`、`claude-opus-4-8`、`claude-fable-5`），Sonnet 5 促销价按日期切换或直接标注两档；顺手把注释里"理想接 claude-api skill 的权威单价"落实——skill 的 Current Models 表就是权威源。**工作量 S，收益最大，建议第一个做。**

### A2. 配置分层模型失真（2.1.207 语义变化）

现状：`shared/types/settings.ts:1`、`electron/services/file-manager-base.ts:16-22` 只建模 user / project / local 三层，`settings-writer.ts` 的 `getEffective`（:123）对**所有键**按 user < project < local 合并。

2.1.207 起两个键的来源规则变了：
- `autoMode` 不再从 `.claude/settings.local.json`（仓库内）读取——只认用户级；
- `pluginConfigs` 不再从项目级 `.claude/settings.json` 读取——只认 user / `--settings` / managed。

后果：CC Harness 对这两个键会显示错误的"生效来源"——用户在 local/project 里写了 `autoMode`，工具说"生效"，实际 Claude Code 根本不读。"帮你看清配置为什么这样生效"是产品核心卖点，这里错了伤根本。

修法建议：给合并逻辑加 per-key 层掩码（键 → 允许来源集合），UI 上被排除的来源标灰并注明"自 2.1.207 起 Claude Code 不再读取此层的该键"。顺带记录：**managed settings（企业管控层）完全没建模**——2.1.207 还修了 remote managed settings 在非交互模式被静默 consent 的问题，说明这层的存在感在上升，对照表建议加行（见 B12）。工作量 M。

### A3. 权限页三处过时

现状：`shared/permission/parse.ts`、`electron/services/file-manager-permissions.ts:13-52`、`src/pages/Permissions.tsx`。

1. **"always allow" 存储位变了（2.1.211）**：批准规则现在存仓库根，跨 worktree 生效。现状只读 `<cwd>/.claude/settings.local.json`（`file-manager-base.ts:16-22`）——cwd 在 worktree 里时会漏读仓库根的规则。注意这不是边角场景：后台 agent 默认就在 worktree 里跑。修法：来源解析加"仓库根"一层（主 checkout 的 git toplevel）。
2. **权限模式零建模**：`default`(→`manual`)/`plan`/`acceptEdits`/`bypassPermissions`/auto mode 在代码里完全不存在（grep 确认）。2.1.200 改名 Manual、2.1.203 给 footer 加 ⏸ 徽标，官方在把"当前模式"变成一等公民信息。低成本切入点：`shared/types/session.ts:70` 注释里列了 `permission-mode` meta 记录类型但目前落进 meta 被丢弃——session 解析层捞出来，监视器就能显示"这个会话当时在什么权限模式下跑"，观测面白捡一块。
3. **规则构造器会引导用户写官方已告警的规则（2.1.210）**：`Write(path)`/`NotebookEdit(path)`/`Glob(path)` 权限规则现在启动时告警，官方建议改用 `Edit(path)`/`Read(path)`。`TOOL_PARAM_HINTS`（`parse.ts:8-15`）和 `KNOWN_TOOLS` 选择器包含 Write——构造器应在用户选择这些工具时同步提示。

工作量合计 M。

### A4. MCP 配置路径与当前 Claude Code 脱节（存量偏差，越拖越大）

现状：`electron/services/file-manager-mcp.ts:7-38` 读 `~/.claude/claude_mcp_config.json`（+legacy `~/.claude/mcpServers.json`）和 `<cwd>/.claude/mcpServers.json`。**不读标准的项目级 `.mcp.json`，不读 `~/.claude.json` 里的 mcpServers**。这不是 2.1.2xx 引入的，是存量偏差，但 MCP 配置面在持续扩大（2.1.186 `claude mcp login/logout`、2.1.206 每 server `request_timeout_ms` 真正生效、2.1.187 工具调用 5 分钟空闲超时 `CLAUDE_CODE_MCP_TOOL_IDLE_TIMEOUT`），路径不对，后面每个新特性都接不上。

配套两条：
- `shared/types/mcp.ts` 的 `timeout` 字段与官方 `request_timeout_ms` 命名对不上，2.1.206 起后者是真实生效的配置键。
- **安全姿态对齐（建议认真考虑）**：`electron/services/mcp/mcp-prober.ts` 的健康检查会真实 spawn stdio server 发 `initialize`。2.1.196 官方刚堵掉"`claude mcp list` 自动拉起仓库自批 server"的供应链洞——同一个威胁模型对 CC Harness 成立：用户拿它打开一个不熟的克隆仓库，健康面板不应自动拉起项目级声明的 server。建议对不受信工作区的项目级 stdio server 默认显示"待确认"，用户点头才 probe。

工作量 M-L。
