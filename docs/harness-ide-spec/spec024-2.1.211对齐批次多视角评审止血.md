把 2.1.211 对齐四批次（价格表、配置分层、权限仓库根、MCP 标准路径、后台 daemon 观测面、拓扑嵌套等）做一次 **10 视角对抗式 code-review**，并把其中 CONFIRMED 的自引入缺陷全部止血。核心教训写在最前面：**这批代码全部通过了 tsc strict + 真机冒烟，仍藏着两个逻辑级真 bug（拓扑 join 键、MCP 门控自穿）——"类型过、跑得动"不等于"逻辑对"，多视角评审是必要的一道闸。**

评审方法（可复用）：5 个正确性视角（逐行 / 删除行为 / 跨文件 / 语言陷阱 / 包装合并）+ 3 个清理视角（复用 / 简化 / 效率）+ 深度视角 + CLAUDE.md 约定视角，各自独立出候选、互不抑制；去重后逐条对**真实代码与落盘数据**核实为 CONFIRMED / PLAUSIBLE / REFUTED。同一缺陷被 4-6 个视角独立命中即视为强验证。

## 现状（评审前的缺陷，引真实 file:line）

### 1. 拓扑 join 键彻底错（最高严重，6 视角命中）

`electron/services/session/agent-topology.ts` 的 `plainSubagentNodes` 用 `byId.get(agentId)` 关联落盘文件与主会话 Task 节点，其中 `agentId` = 文件名 hex（`agent-<hex>.jsonl`），而 taskTree 节点的 id 是 tool_use id（`toolu_...`）。二者不同 id 空间，**永远匹配不上**。真实关联键在同名 `.meta.json` 的 `toolUseId` 字段，旧代码只读了 `agentType` 没读它。

真机实证：`agent-a00a490d97d455b23.meta.json` 内容 `{"agentType":"general-purpose","toolUseId":"toolu_01EPgMpc..."}`——文件名 hex `a00a...` ≠ `toolu_...`。

后果：(1) 充实分支（token/时长/filePath）永远不命中，Task 节点点开无抽屉回放；(2) 每个 subagent 落到"未匹配"分支被二次建节点 → 拓扑图每个 subagent 渲染两次、token 双算、`agentTopologyLayout.ts:56` 的重复 node id 被 reactflow 静默丢弃；(3) `depth>=5` 护栏是死代码（所有文件派生节点 depth 恒 0，宣称的五层嵌套从未实现）。

### 2. MCP 供应链门控被自己绕过（高严重，6 视角命中）

`electron/services/file-manager-mcp.ts`：`getMCPServers` 合并 5 个来源（含 legacy `<project>/.claude/mcpServers.json`），但 `getMCPServerSources` 只覆盖标准路径 + `~/.claude.json`，漏了两个 legacy 路径。只存在于 legacy 项目文件的 stdio server 拿不到 `project` 归属 → `electron/ipc/mcp.ts` 的 `deferStdio` 集（按 `source==='project'` 构建）漏掉它 → 批量健康检查 `probeMCP` 自动 spawn 仓库控制的可执行文件，**恰好穿透这批刚加的 2.1.196"待确认"门控**。

### 3. MCP 删除/保存对 `~/.claude.json` 静默无效 / 分叉

`deleteMCPServer` 从不写 `~/.claude.json`（CLI 管理、本工具只读），但 `getMCPServers` 合并它——删除只存在于该文件的 server 静默无效，刷新后又回来；编辑则把它分叉进 `claude_mcp_config.json`，本工具显示新配置而 CLI 仍读旧的。`src/pages/MCP.tsx` 的 `save`/`remove` 无 try/catch，抛错会变未处理 rejection。

### 4. 权限：worktree 里删仓库根规则静默无效

`electron/services/file-manager-permissions.ts` `getPermissionModel` 把仓库根 `settings.local.json` 规则并入 local 视图（2.1.211），但 `deletePermissionRule` 只写 `projectPath` 的本地文件——worktree 里删仓库根规则 filter 掉的是不存在的项、写回不变、下次合并又回来，且 UI 无法察觉规则实际在仓库根文件。

### 5. 价格 promo 模块加载时定死 + 旧 Opus 漏价

`shared/data/model-pricing.ts`：`SONNET5_PROMO_ACTIVE = Date.now() < ...` 在模块加载时求值一次并烘进 TABLE → 长驻 Electron 进程跨过 2026-09-01 后永远算 $2/$10。且通用 `/opus/ → $5/$25` 对旧 Opus（4.1/4.0/Opus 3，实际 $15/$75）算低三倍。

### 6. 其余 CONFIRMED

- `shared/hooks/matcher-semantics.ts`：逗号判定先于正则判定 → 含逗号的正则（`{0,20}`、字符类）被误判成逗号列表并谎报"逐个精确匹配"，与 CLI 实际相反。
- `electron/services/settings-writer.ts` `levelHonoredForKey` / `shared/data/settings-key-catalog.ts` `settingsKeyMeta`：`OBJ[key]` 未防原型链 → 键名为 `constructor`/`toString` 时取到函数、`.includes` 抛错、整个 Settings 页崩。
- `settings-writer.ts` `getModel`：全落在不生效层时 winner 层既作 source 又进 ignored、其余层在 overridden 与 ignored 双列。
- `electron/services/bg-agents-service.ts` `resolveClaudeBin`：只有 POSIX 候选路径且 `execFile` 跑不了 Windows 的 `claude.cmd` shim → 整个后台页在 Windows 死掉并误报 CLI 版本问题；`readJson` 吞掉损坏文件的错误无日志。
- `src/pages/BgAgents.tsx` `groupOf`/`failed`：`item.job?.state ?? item.state` 让过期磁盘 state 压过 CLI 权威 → 已结束 job 误报"干活中"（实证：CLI `failed` vs 磁盘 `working`）；React key `${sessionId}:${pid??id??''}` 可碰撞；`copyAttach` setTimeout 无卸载清理。

- **未纳入（PLAUSIBLE / 非本 spec 范围）**：MCP 5 文件读虽已并行但 `~/.claude.json`（可多 MB）仍每次全解析——已并行化，mtime 缓存留作后续效率项；`bg-agents` 在 CLI 调用失败时丢弃磁盘 roster/state（可做降级渲染）留作增强；拓扑真实祖孙嵌套（depth>1）已支持但缺真实多层样本回归。
